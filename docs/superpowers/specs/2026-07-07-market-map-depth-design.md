# Stock Papi 產業地圖深化設計

## 目標

在不增加 Cloud Run 即時計算、外部 API 或前端套件的前提下，讓產業地圖從清單升級為可快速判讀的市場總覽，並保留 Stock Papi 現有明亮品牌與 LINE／Web 分工。

## 採用方案

擴充既有本地 `market-insights` 快照。從每檔既有 artifact 讀取 AI 機率、單日報酬、法人比率、融資變化、融券變化與量能，於本地彙總後交給 Cloud Run 顯示。缺欄位一律降級，不補抓、不猜測。

## 頁面結構

1. 產業關鍵指標：領先產業、領先個股、多頭廣度與資料覆蓋。
2. 產業漲跌熱力圖：用產業平均單日報酬決定顏色，用有效樣本數決定區塊大小；每格顯示產業、平均漲跌、平均 AI 機率與樣本數。
3. 籌碼訊號：只顯示現有資料可支持的法人、融資與量能，不仿造投信／大戶等缺乏一致來源的欄位。
4. 產業明細：每個產業顯示前五名公司、價格、單日漲跌、AI 機率、趨勢與可解釋標籤。
5. 台美日供應鏈：沿用既有角色分類，補上可取得的 AI 機率、漲跌與訊號；不宣稱未驗證的供應契約。
6. MOPS 與 ETF 持倉維持原有區塊。

網站不新增收藏按鈕；關注與提醒仍由 LINE 管理。

## 資料與錯誤處理

- 所有新增數值由 `D:\StockPapiData` 既有 artifact 產生。
- 單一股票缺少欄位時只省略該值；產業無有效樣本時不進熱力圖。
- 快照未上傳、過期或雜湊驗證失敗時，沿用目前降級頁，不在 Cloud Run 補算。
- 既有 schema version 保持不變；新增欄位向後相容。

## 測試

- 純函式測試產業平均值、排序、熱度級別與缺漏資料。
- Web route 測試關鍵指標、熱力圖、籌碼訊號、產業明細與供應鏈標籤。
- 完整測試、Python 編譯、JavaScript 語法與 `git diff --check`。

## 無資料恢復規則

- 洞察物件必須先於全市場股票物件上傳，避免大型上傳超時時整張地圖沒有資料。
- 降級模式只使用八個正式主題產業，不顯示「全市場」或「ETF專區」。
- 每個正式主題至少配置五檔公司；缺少量化值時仍顯示公司名稱與「等待更新」，不得製造機率或漲跌。
- `coverage` 只計算具有有效 AI 機率的公司，不能把等待更新的公司算成有效樣本。
